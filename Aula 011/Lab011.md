## Cenário: Provisionamento de Armazenamento e Banco de Dados
Vamos simular o provisionamento dos dois serviços de dados mais críticos usando a linha de comando.

## Seção 1: Gerenciamento de Arquivos no S3 (Armazenamento de Objetos)
Vamos criar um *bucket* para armazenar arquivos de configuração e provar o *upload* a partir do WSL.

1.  Crie o diretório de trabalho do laboratório.
```bash
mkdir -p ~/aulas_lab/aula011
cd ~/aulas_lab/aula011
```

2.  **Criação do Arquivo Local:** Crie um arquivo de configuração de teste no seu terminal WSL.
```bash
touch config_app_v1.txt
echo "DB_HOST=localhost" > config_app_v1.txt
```

3.  **Definir Nome do Bucket:** O nome deve ser **globalmente único** e em minúsculas. Use seu nome ou um identificador único.
```bash
BUCKET_NAME="lab-devops-12-unifaat" 
```

3.  **Criar o Bucket:** Crie o *bucket* na sua região padrão.
```bash
aws s3 mb s3://$BUCKET_NAME
```

4.  **Upload do Arquivo:** Copie o arquivo local para o *bucket* usando o comando `cp`.
```bash
aws s3 cp config_app_v1.txt s3://$BUCKET_NAME/config/
```

5.  **Verificação:** Liste o conteúdo do *bucket* (e o subdiretório `config/`) para confirmar o *upload*.
```bash
aws s3 ls s3://$BUCKET_NAME/config/
```

6.  **Limpeza do S3:** Remova o arquivo e depois o *bucket* (o *bucket* deve estar vazio para ser removido).
```bash
aws s3 rm s3://$BUCKET_NAME/config/config_app_v1.txt
aws s3 rb s3://$BUCKET_NAME
```

## Seção 2: Simulação de Criação de RDS (Database Gerenciado)
A criação de um RDS real é complexa e demorada. Usaremos o comando `aws rds describe-db-instances` para simular a inspeção e entender o potencial da CLI.

1.  **Pré-requisito:** Obtenha o ID de uma **VPC** existente na sua conta, pois o RDS precisa ser lançado dentro de uma VPC.
```bash
VPC_ID=$(aws ec2 describe-vpcs --query "Vpcs[0].VpcId" --output text)
echo "VPC ID Selecionada: $VPC_ID"
```

2.  **Simulação da Criação (Comando Longo):** **Não execute este comando**, apenas inspecione a complexidade. Este comando iniciaria um DB MySQL gerenciado (Micro, Dev/Test).
```bash
# Criando um Cluster Aurora
aws rds create-db-cluster \
  --db-cluster-identifier devopsdb \
  --engine aurora-postgresql \
  --engine-version 17.4 \
  --master-username postgres \
  --master-user-password UniFAAT2026 \
  --vpc-security-group-ids sg-0562852fe8fe3cf92 \
  --db-subnet-group-name default \
  --backup-retention-period 7 \
  --preferred-backup-window "09:58-10:28" \
  --preferred-maintenance-window "mon:04:31-mon:05:01" \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:us-east-2:353818015911:key/a5ca8923-7297-4dc9-9f9b-17b137886162 \
  --region us-east-2

# Criar a Instancia do DB RDS Aurora
aws rds create-db-instance \
  --db-instance-identifier devopsdb-instance-1 \
  --db-instance-class db.serverless \
  --engine aurora-postgresql \
  --db-cluster-identifier devopsdb \
  --availability-zone us-east-2c \
  --publicly-accessible \
  --monitoring-interval 60 \
  --monitoring-role-arn arn:aws:iam::353818015911:role/rds-monitoring-role \
  --enable-performance-insights \
  --performance-insights-retention-period 7 \
  --performance-insights-kms-key-id arn:aws:kms:us-east-2:353818015911:key/a5ca8923-7297-4dc9-9f9b-17b137886162 \
  --auto-minor-version-upgrade \
  --no-deletion-protection \
  --region us-east-2

# Verificar o Status de Criação
aws rds describe-db-instances --db-instance-identifier devopsdb-instance-1 --region us-east-2

```
*(**Conclusão:** Observe como o comando `create-db-instance` encapsula a imagem (engine), o tipo de instância (class), o volume (storage), a persistência (identifier) e a rede (vpc-security-group-ids) em uma única linha.)* 

3.  **Inspeção (Se existirem DBs):** Se você já possui alguma instância RDS em sua conta, inspecione seus detalhes.
```bash
aws rds describe-db-instances \
--query 'DBInstances[0].[DBInstanceIdentifier, DBInstanceStatus, Endpoint.Address]'
```
*(**Resultado Esperado:** Visualizar o identificador, o status e o endereço de *endpoint* do DB — a informação que sua aplicação precisaria para se conectar.)*

## Seção 3: Conectando ao RDS via Terminal

1. Instalar cliente PostgreSQL (se ainda não estiver instalado)
sudo apt update
sudo apt install postgresql-client -y

1. Conectar ao banco usando psql
*Substitua os valores entre <> pelos dados do seu RDS*

```bash
psql \
    --host=devopsdb-instance-1.c3mw8u2sgtjh.us-east-2.rds.amazonaws.com \
    --port=5432 \
    --username=postgres \
    --password \
    --dbname=postgres
```
> Comandos úteis dentro do psql:
> \l - listar databases
> \c nome_database - conectar a um database específico
> \dt - listar tabelas
> \q - sair do psql

> Exemplo de conexão usando string de conexão completa
> psql "postgresql://postgres:UniFAAT2026@devopsdb-instance-1.xxxxx.us-east-2.rds.amazonaws.com:5432/postgres"

2. Para executar queries diretamente do terminal sem entrar no modo interativo
```bash
psql \
    --host=devopsdb-instance-1.c3mw8u2sgtjh.us-east-2.rds.amazonaws.com \
    --port=5432 \
    --username=postgres \
    --password \
    --dbname=postgres \
    -c "SELECT version();"
```
## Seção 4: Limpeza
1. Primeiro, apague a instância do banco de dados:
```bash
aws rds delete-db-instance \
  --db-instance-identifier devopsdb-instance-1 \
  --skip-final-snapshot \
  --delete-automated-backups \
  --region us-east-2
```

2. Aguarde a instância ser completamente removida, depois apague o cluster Aurora:
```bash
aws rds delete-db-cluster \
  --db-cluster-identifier devopsdb \
  --skip-final-snapshot \
  --delete-automated-backups \
  --region us-east-2
```

3. Apagar pastas
```bash
cd ~
rm -rf ~/aulas_lab/aula011
```

## Dicas Importantes

1. **S3 é object storage** - ideal para arquivos estáticos e backups
2. **RDS é gerenciado** - AWS cuida de patches, backups e alta disponibilidade
3. **Nomes de bucket são globais** - devem ser únicos em toda a AWS
4. **Use VPC e security groups** - sempre configure segurança adequada
5. **Monitore custos** - RDS e S3 geram custos, sempre limpe recursos de teste
