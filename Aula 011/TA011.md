## 1. Conceitos Fundamentais

### Tipos de Armazenamento na AWS

A AWS oferece diferentes tipos de armazenamento para diferentes necessidades. É como ter uma garagem (block storage), um depósito (object storage) e um arquivo (file storage) - cada um serve para propósitos específicos.

#### Block Storage vs Object Storage vs File Storage

**Block Storage (EBS)**:
- Armazenamento de baixo nível em blocos
- Anexado a instâncias EC2 como discos
- Performance consistente e baixa latência
- Ideal para: OS, bancos de dados, file systems

**Object Storage (S3)**:
- Armazenamento de objetos com metadados
- Acesso via API REST/HTTP
- Escalabilidade virtualmente ilimitada
- Ideal para: backups, websites estáticos, data lakes

**File Storage (EFS)**:
- Sistema de arquivos compartilhado
- Acesso simultâneo de múltiplas instâncias
- Protocolo NFS (Network File System)
- Ideal para: aplicações distribuídas, content management

### Amazon S3 (Simple Storage Service)

#### Conceitos Fundamentais

**Buckets**: Containers para objetos
- Nome globalmente único
- Região específica
- Até 100 buckets por conta (padrão)

**Objects**: Arquivos armazenados no S3
- Tamanho: 0 bytes até 5TB
- Chave (key): Nome único dentro do bucket
- Metadados: Informações adicionais sobre o objeto

```bash
# Estrutura hierárquica simulada
s3://meu-bucket/
├── fotos/
│   ├── 2024/janeiro/foto1.jpg
│   └── 2024/fevereiro/foto2.jpg
├── documentos/
│   └── relatorio.pdf
└── logs/
    └── app.log
```

#### Classes de Armazenamento S3

**S3 Standard**:
- Uso geral, acesso frequente
- Durabilidade: 99.999999999% (11 9's)
- Disponibilidade: 99.99%
- Custo: Mais alto, mas sem taxas de recuperação

**S3 Standard-IA (Infrequent Access)**:
- Acesso menos frequente, mas rápido quando necessário
- Custo de armazenamento menor
- Taxa de recuperação por GB
- Ideal para: backups, disaster recovery

**S3 One Zone-IA**:
- Armazenado em apenas uma AZ
- 20% mais barato que Standard-IA
- Menor durabilidade (99.999999999% em uma AZ)
- Ideal para: dados replicáveis, backups secundários

**S3 Glacier**:
- Arquivamento de longo prazo
- Tempo de recuperação: minutos a horas
- Muito baixo custo
- Ideal para: arquivos históricos, compliance

**S3 Glacier Deep Archive**:
- Arquivamento de muito longo prazo
- Tempo de recuperação: 12 horas
- Menor custo de armazenamento
- Ideal para: dados acessados raramente (7-10 anos)

### Amazon EBS (Elastic Block Store)

#### Tipos de Volume EBS

**gp3 (General Purpose SSD)**:
- Última geração, melhor custo-benefício
- IOPS e throughput configuráveis independentemente
- Baseline: 3,000 IOPS, 125 MB/s
- Máximo: 16,000 IOPS, 1,000 MB/s

**gp2 (General Purpose SSD)**:
- Geração anterior
- IOPS baseado no tamanho (3 IOPS por GB)
- Burst até 3,000 IOPS para volumes pequenos
- Máximo: 16,000 IOPS

**io2/io1 (Provisioned IOPS SSD)**:
- Alta performance para workloads críticos
- IOPS provisionado independente do tamanho
- io2: até 64,000 IOPS, 99.999% durabilidade
- io1: até 64,000 IOPS, 99.999% durabilidade

**st1 (Throughput Optimized HDD)**:
- Otimizado para throughput sequencial
- Até 500 MB/s
- Ideal para: big data, data warehouses, log processing

**sc1 (Cold HDD)**:
- Menor custo
- Até 250 MB/s
- Ideal para: workloads com acesso infrequente

#### Snapshots EBS
```bash
# Criar snapshot
aws ec2 create-snapshot \
  --volume-id vol-1234567890abcdef0 \
  --description "Backup diário"

# Listar snapshots
aws ec2 describe-snapshots --owner-ids self

# Criar volume a partir de snapshot
aws ec2 create-volume \
  --snapshot-id snap-1234567890abcdef0 \
  --availability-zone us-east-1a
```

### Amazon EFS (Elastic File System)

#### Características do EFS
- **POSIX-compliant**: Sistema de arquivos padrão Linux
- **Escalabilidade**: Cresce automaticamente até petabytes
- **Concurrent Access**: Milhares de conexões simultâneas
- **Performance**: General Purpose ou Max I/O
- **Throughput**: Provisioned ou Bursting

```bash
# Montar EFS em EC2
sudo mount -t efs fs-12345678:/ /mnt/efs

# Entrada no /etc/fstab para mount automático
fs-12345678.efs.us-east-1.amazonaws.com:/ /mnt/efs efs defaults,_netdev 0 0
```

### Bancos de Dados AWS

#### Amazon RDS (Relational Database Service)

**Engines Suportadas**:
- **MySQL**: Open source, amplamente usado
- **PostgreSQL**: Open source, recursos avançados
- **MariaDB**: Fork do MySQL
- **Oracle**: Enterprise, licenciamento próprio
- **SQL Server**: Microsoft, licenciamento próprio
- **Amazon Aurora**: MySQL/PostgreSQL compatível, performance superior

**Benefícios do RDS**:
- **Gerenciamento**: Patches, backups, monitoring automáticos
- **Escalabilidade**: Vertical (instance type) e horizontal (read replicas)
- **Alta Disponibilidade**: Multi-AZ deployment
- **Segurança**: Encryption, VPC, IAM integration
- **Backup**: Automated backups, point-in-time recovery

```bash
# Criar instância RDS
aws rds create-db-instance \
  --db-instance-identifier mydb \
  --db-instance-class db.t3.micro \
  --engine mysql \
  --master-username admin \
  --master-user-password mypassword \
  --allocated-storage 20 \
  --vpc-security-group-ids sg-12345678
```

#### Amazon Aurora

**Vantagens do Aurora**:
- **Performance**: 5x MySQL, 3x PostgreSQL
- **Durabilidade**: 6 cópias em 3 AZs
- **Auto Scaling**: Storage cresce automaticamente
- **Serverless**: Escala automaticamente baseado na demanda
- **Global Database**: Replicação cross-region com latência <1s

**Aurora Serverless**:
```bash
# Criar cluster Aurora Serverless
aws rds create-db-cluster \
  --db-cluster-identifier aurora-serverless \
  --engine aurora-mysql \
  --engine-mode serverless \
  --master-username admin \
  --master-user-password mypassword \
  --scaling-configuration MinCapacity=2,MaxCapacity=16,AutoPause=true
```

#### Amazon DynamoDB

**Características do DynamoDB**:
- **NoSQL**: Banco de dados de documentos e chave-valor
- **Serverless**: Totalmente gerenciado
- **Performance**: Latência de milissegundos
- **Escalabilidade**: Automática baseada na demanda
- **Global Tables**: Replicação multi-region

**Estrutura de Dados**:

### Estratégias de Backup e Disaster Recovery

#### RTO vs RPO
- **RTO (Recovery Time Objective)**: Tempo máximo para restaurar serviço
- **RPO (Recovery Point Objective)**: Perda máxima de dados aceitável

#### Estratégias de DR

**Backup and Restore** (RTO: horas, RPO: horas):
- Backups regulares para S3/Glacier
- Restauração quando necessário
- Menor custo, maior tempo de recuperação

**Pilot Light** (RTO: 10s de minutos, RPO: minutos):
- Infraestrutura mínima sempre rodando
- Dados replicados continuamente
- Scale up quando necessário

**Warm Standby** (RTO: minutos, RPO: minutos):
- Versão reduzida do ambiente em outra região
- Sempre rodando, mas com capacidade menor
- Scale up rápido quando necessário

**Multi-Site Active/Active** (RTO: segundos, RPO: segundos):
- Ambientes completos em múltiplas regiões
- Tráfego distribuído entre sites
- Maior custo, menor tempo de recuperação

### Otimização de Custos

#### S3 Cost Optimization
```bash
# S3 Intelligent Tiering
aws s3api put-bucket-intelligent-tiering-configuration \
  --bucket meu-bucket \
  --id EntireBucket \
  --intelligent-tiering-configuration Id=EntireBucket,Status=Enabled,Filter={}

# Análise de custos por storage class
aws s3api get-bucket-analytics-configuration \
  --bucket meu-bucket \
  --id analytics-config
```

#### RDS Cost Optimization
- **Reserved Instances**: 1-3 anos, até 60% desconto
- **Right Sizing**: Monitorar CPU/Memory utilization
- **Aurora Serverless**: Para workloads variáveis
- **Read Replicas**: Distribuir carga de leitura

## 2. Questões de Múltipla Escolha

### Questão 1
Qual classe de armazenamento S3 é mais adequada para dados que são acessados raramente, mas quando necessário, precisam ser recuperados rapidamente?

a) S3 Standard
b) S3 Standard-IA (Infrequent Access)
c) S3 Glacier
d) S3 Glacier Deep Archive

**Gabarito: B**

**Justificativa:** S3 Standard-IA é projetado exatamente para esse cenário - dados acessados com pouca frequência, mas que quando necessário, devem estar disponíveis imediatamente. Oferece menor custo de armazenamento que S3 Standard, mas cobra taxa de recuperação. S3 Glacier e Deep Archive têm tempos de recuperação de minutos a horas, não sendo adequados para acesso rápido.

### Questão 2
Qual é a principal diferença entre Amazon RDS e Amazon Aurora?

a) RDS é para bancos relacionais, Aurora é NoSQL
b) Aurora oferece melhor performance e recursos específicos da AWS
c) RDS só funciona com MySQL, Aurora com PostgreSQL
d) Aurora é mais barato que RDS

**Gabarito: B**

**Justificativa:** Amazon Aurora é um engine de banco de dados desenvolvido pela AWS que oferece performance superior (5x MySQL, 3x PostgreSQL), recursos específicos como auto-scaling de storage, Aurora Serverless, e Global Database. Ambos são relacionais e suportam MySQL/PostgreSQL. Aurora geralmente custa mais, mas oferece melhor performance e recursos avançados.

### Questão 3
Em uma estratégia de Disaster Recovery, qual abordagem oferece o menor RTO (Recovery Time Objective)?

a) Backup and Restore
b) Pilot Light
c) Warm Standby
d) Multi-Site Active/Active

**Gabarito: D**

**Justificativa:** Multi-Site Active/Active oferece o menor RTO (segundos) pois mantém ambientes completos rodando simultaneamente em múltiplas regiões com tráfego distribuído. Se um site falha, o tráfego é imediatamente redirecionado. As outras estratégias requerem tempo para provisionar recursos (Backup and Restore: horas, Pilot Light: 10s de minutos, Warm Standby: minutos).
